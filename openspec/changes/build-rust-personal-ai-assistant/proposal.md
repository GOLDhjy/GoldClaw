## Why

需要一个基于 Rust 的个人 AI 助手基础设施，它既能像 openclaw 一样承担本地 AI 助手职责，又要在工程上具备更清晰的分层、可后台常驻的网关能力、可恢复的初始化流程，以及可诊断的运行状态。现有个人助手项目通常把运行时、界面和渠道接入耦合在一起，导致跨终端复用、后台稳定运行、问题排查和多渠道接入成本都偏高，因此需要先建立一套可持续演进的底座。

## What Changes

- 建立 Rust workspace 级别的整体架构，把运行时、网关守护进程、配置/状态、诊断、客户端界面和渠道适配器拆分为独立模块。
- 增加一个可后台运行的本地网关/守护进程，统一负责会话、模型访问、工具调用、渠道路由、事件分发和生命周期管理。
- 增加清晰的初始化引导流程，包括首次配置、Provider 选择、凭据录入、服务启动、渠道启用和验证步骤。
- 增加 `doctor` 诊断体系，对配置、密钥、网络、Provider、端口、Webhook、数据库和渠道健康状态进行检查，并给出修复建议。
- 提供统一能力下的多前端入口，包括 CLI/TUI/Web 三种一方客户端，保证核心会话行为一致。
- 建立渠道接入框架，并提供飞书、企业微信、微信三类参考连接器能力与消息收发链路。
- 增加日志、追踪、事件审计、配置版本化、SQLite 本地状态持久化和服务安装/卸载等运维基础能力。

## Capabilities

### New Capabilities
- `assistant-runtime-core`: 统一的 AI 助手运行时，负责会话编排、模型/工具调用、状态持久化接口和事件总线。
- `gateway-daemon`: 可后台运行的本地网关与守护进程，负责本地 API、事件流、生命周期管理和多入口路由。
- `setup-and-doctor`: 初始化向导、配置引导、环境检查、健康诊断和修复建议输出。
- `multi-surface-clients`: 面向 CLI、TUI、Web 的统一客户端体验和共享会话访问模型。
- `channel-connectors`: 飞书、企业微信、微信等渠道连接器框架，以及统一的入站/出站消息标准。

### Modified Capabilities

None.

## Impact

- 新增 Rust workspace crate 划分、守护进程入口、客户端入口和连接器模块。
- 引入本地 HTTP/WebSocket、跨平台 IPC、SQLite、本地配置文件、密钥存储和日志追踪依赖。
- 定义本地控制面 API、客户端会话 API、渠道消息标准和健康检查输出格式。
- 增加渠道 webhook/polling 处理、消息签名校验、重试与故障隔离等基础设施。
